# What are Declaration Files?

When you use a JavaScript library in TypeScript, the compiler needs to know the **shapes**
of the values that library exports — function signatures, class structures, constant types.
**Declaration files** (`.d.ts` files) provide exactly this: type information without any
runtime code. They are the bridge between untyped JavaScript and TypeScript's type system.

## Core Concept

A declaration file contains only **type declarations** — no executable code. It uses the
`declare` keyword to tell TypeScript "this value exists at runtime, trust me on its shape."

```typescript
// math-lib.d.ts — declares types for a JavaScript library
declare function add(a: number, b: number): number;
declare function subtract(a: number, b: number): number;
declare const PI: number;

declare interface MathConfig {
    precision: number;
    roundingMode: "ceil" | "floor" | "round";
}
```

The `declare` keyword means "this exists somewhere else — I'm just describing its type."
Nothing in a `.d.ts` file produces JavaScript output. When TypeScript compiles your project,
`.d.ts` files are consumed for type checking but never emitted as `.js`.

### Where Declaration Files Come From

Declaration files appear in three ways:

1. **Generated by `tsc`** — When you compile a TypeScript library with `"declaration": true`
   in `tsconfig.json`, the compiler emits `.d.ts` files alongside the `.js` output.
2. **Bundled with a package** — Many npm packages ship their own `.d.ts` files (look for
   a `"types"` or `"typings"` field in `package.json`).
3. **Written manually** — For untyped JavaScript libraries, you write `.d.ts` files yourself
   or install community-maintained types from DefinitelyTyped (`@types/*`).

## How It Works

### The `declare` Keyword

`declare` introduces an **ambient declaration** — a type description for something that
exists at runtime but isn't defined in the current TypeScript compilation:

```typescript
// Ambient variable — exists globally at runtime
declare const API_URL: string;

// Ambient function — provided by a script tag or bundler
declare function analytics(event: string, data: Record<string, unknown>): void;

// Ambient class — from a JavaScript library
declare class EventEmitter {
    on(event: string, listener: (...args: unknown[]) => void): this;
    emit(event: string, ...args: unknown[]): boolean;
    removeListener(event: string, listener: (...args: unknown[]) => void): this;
}

// Ambient enum — values exist at runtime
declare enum LogLevel {
    Debug = 0,
    Info = 1,
    Warn = 2,
    Error = 3,
}
```

### Ambient Modules

You can declare types for an entire module using `declare module`:

```typescript
// custom-lib.d.ts
declare module "custom-lib" {
    export function parse(input: string): object;
    export function stringify(data: object): string;
    export const version: string;
}
```

Now any file can `import { parse } from "custom-lib"` and TypeScript knows the types.

### Wildcard Module Declarations

For non-code imports (CSS, images, JSON), use wildcard patterns:

```typescript
// assets.d.ts
declare module "*.css" {
    const classes: Record<string, string>;
    export default classes;
}

declare module "*.png" {
    const src: string;
    export default src;
}
```

This tells TypeScript what shape to expect when you `import styles from "./app.css"`.

## Common Pitfalls

- **Forgetting `declare` in `.d.ts` files.** In a `.d.ts` file, top-level declarations
  are automatically ambient, so `declare` is optional. But in a regular `.ts` file, you
  must use `declare` explicitly for ambient declarations.
- **Putting executable code in `.d.ts` files.** Declaration files cannot contain
  implementations — no function bodies, no variable assignments with values, no class
  method bodies. Only type signatures.
- **Conflicting declarations.** If you declare a module that already has types (e.g., from
  `@types/*`), your declaration may conflict. Use module augmentation instead of
  redeclaring the entire module.
- **Missing `export` in module declarations.** Inside `declare module "x" { ... }`, you
  must `export` the declarations you want consumers to see. Without `export`, they're
  module-private.

## Key Takeaways

- Declaration files (`.d.ts`) contain only type information — no runtime code.
- The `declare` keyword creates ambient declarations for values that exist elsewhere.
- `declare module "name"` provides types for an entire module.
- Wildcard module declarations (`declare module "*.css"`) handle non-TypeScript imports.
- TypeScript can auto-generate `.d.ts` files with the `"declaration": true` compiler option.
- Declaration files are the foundation for typing the entire JavaScript ecosystem.

<div class="hint">
You can check whether a package includes its own types by looking at its `package.json`
for a `"types"` or `"typings"` field. If it points to a `.d.ts` file, the package is
self-typed. If not, check for `@types/package-name` on npm.
</div>

<div class="hint">
When TypeScript encounters `import { x } from "some-lib"`, it searches for types in this
order: the package's own `"types"` field, then `@types/some-lib`, then any `declare module`
in your project. Understanding this resolution order helps debug "cannot find module" errors.
</div>
